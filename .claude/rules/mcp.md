# MCP & Azure DevOps Tooling Rules — ABSOLUTE RULES

These rules apply to ALL MCP server code, MCP tool implementations, the Azure DevOps API client layer, transport, and the tool code-generation pipeline. They are **VERY STRICT and ABSOLUTELY NON-NEGOTIABLE**!
General Rust idioms, error handling, testing, and the Definition of Done live in `rust.md`. Project structure, invariants, configuration, and Make targets live in `project.md`. Behavioral/process/git/plan rules live in `agent.md`.

## 1) MCP Tool Conventions — ABSOLUTE RULES

- Each tool lives in its own file under `src/mcp/tools/<category>/`. Tools are isolated and unit-testable.
- Each tool MUST:
  1. Define an `Args` struct deriving `serde::Deserialize` + `schemars::JsonSchema`.
  2. Annotate the async function with `#[mcp_tool(name = "azdo_...", description = "...")]`.
  3. Use the signature `pub async fn <tool_name>(client: &(dyn AzureDevOpsApi + Send + Sync), args: <Args>) -> Result<CallToolResult, McpError>`.
  4. Validate every parameter (type, range, required fields) via serde deserialization and custom deserializers (e.g., `deserialize_non_empty_string` in `support/`).
  5. Convert domain errors (`AzureError`) to `McpError` via `.map_err(...)`.
  6. Build every success result via `tool_text_success(...)` (see §2).
- **Tool naming — ABSOLUTE**: every tool name is prefixed **`azdo_`**. Keep names consistent with `docs/ARCHITECTURE.md` (MCP Tools table).
- **Idempotency**: tool calls MUST be safe to retry.
- Shared per-response utilities live in `src/mcp/tools/support/` (`simplify_work_item_json`, `work_items_to_csv` / `board_columns_to_csv`, `csv_sanitize`, `deserialize_non_empty_string`, `default_text_format`, `tool_text_success`). Reuse them; do NOT duplicate their logic in tool files.

## 2) Anti-Prompt-Injection — ABSOLUTE, ZERO EXCEPTIONS

- EVERY MCP tool, no matter its kind, MUST prepend the anti-prompt-injection warning to its response content. This is NON-NEGOTIABLE.
- The warning is defined as the constant `UNTRUSTED_CONTENT_WARNING` in `src/mcp/tools/support/tool_text_success.rs`.
- ALL tools MUST build their success `CallToolResult` via `tool_text_success()` from `src/mcp/tools/support/`. Calling `CallToolResult::success(vec![Content::text(...)])` directly in tool files is **FORBIDDEN**.
- `tool_text_success()` automatically prepends the `/* ... */` warning comment before the actual content.
- When adding a new MCP tool, you MUST use `tool_text_success()` for ALL success return paths. There are ZERO exceptions.
- You MUST NEVER remove, weaken, or shorten the warning text. If a change to the warning is needed, you MUST discuss it with the user FIRST.

## 3) AzureDevOpsApi Trait Boundary — ABSOLUTE RULES

- All MCP tool functions accept `&(dyn AzureDevOpsApi + Send + Sync)`, NOT `&AzureDevOpsClient`. API calls MUST go through trait methods.
- `AzureDevOpsApi` is defined in `src/azure/api_trait.rs` with `#[cfg_attr(feature = "test-support", mockall::automock)]`.
- `AzureDevOpsClient` implements the trait by delegating to the standalone API functions in the per-area modules (`boards.rs`, `work_items.rs`, etc.), each of which takes `&AzureDevOpsClient` as its first parameter.
- Integration tests use `MockAzureDevOpsApi` to verify tool behavior without external services.
- **Why the `test-support` feature?** `cfg(test)` is NOT active when the library is built as a dependency for integration tests (`tests/*.rs`), so `#[cfg_attr(test, mockall::automock)]` would never generate `MockAzureDevOpsApi`. A Cargo feature ensures the mock is generated whenever tests run. Always use `make test` / `cargo test --features test-support`.

## 4) Tool Code-Generation Pipeline — ABSOLUTE RULES

Tool registration uses a two-phase code-generation approach — do NOT bypass it or hand-write the tool router:

1. **Compile-time (`build.rs`)**: scans `src/mcp/tools/` for `#[mcp_tool(name = "...", description = "...")]` attributes, extracts function signatures, and generates `generated_tools.rs` in `OUT_DIR` — the `#[tool_router] impl AzureMcpServer` block. `src/mcp/server.rs` pulls it in via `include!()`.
2. **Proc-macro (`mcp-tools-codegen`)**: the `#[mcp_tool]` attribute macro validates the `name` + `description` metadata and passes the function through unchanged. The actual routing is generated by `build.rs`.

- Generated code (`generated_tools.rs`) MUST NEVER be hand-edited — change the tool source and regenerate via the build.
- `.unwrap()` is acceptable in `build.rs` (build-time programmer errors), NOT in library/production code.

## 5) Transport & Concurrency — ABSOLUTE RULES

- Two transports: **stdio** (default) and **streamable HTTP** (`--server`). HTTP uses `hyper` + `rmcp::StreamableHttpService` with `LocalSessionManager`, binding `0.0.0.0:<port>`, spawning a tokio task per connection (`src/server/http.rs`).
- `AzureMcpServer` is `Clone` (wraps `Arc<dyn AzureDevOpsApi>` + `ToolRouter<Self>`) and safe to share across tasks; each HTTP connection gets a clone.
- `reqwest::Client` is internally `Arc`'d and connection-pooled — reuse it; NEVER create a new client per request.
- NEVER block in async context — use `tokio::task::spawn_blocking` if unavoidable.
- NEVER spawn fire-and-forget tasks without a clear shutdown path.
- HTTP binds `0.0.0.0` by default — deployment MUST be behind appropriate network controls (see `project.md`).

## 6) Azure DevOps API Integration — ABSOLUTE RULES

- `AzureDevOpsClient` (`src/azure/client.rs`) holds a `reqwest::Client` and an ordered credential chain (`Vec<CredentialSource>`, environment client-secret → Azure CLI → Azure Developer CLI → managed identity); `get_token()` tries each source in order, applying a per-source timeout (only managed identity carries one, 2s, so its IMDS probe fails fast off Azure). Bearer tokens are fetched per-request.
- API modules (`boards.rs`, `work_items.rs`, etc.) are standalone functions taking `&AzureDevOpsClient` as the first parameter, invoked through the HTTP helpers (`get`, `post`, `patch`, `org_request`, `team_request`, `vssps_request`).
- Base URL: `https://dev.azure.com/{organization}/{project}/_apis/`; VSSPS: `https://app.vssps.visualstudio.com/_apis/`.
- API version **7.1** (Comments API `7.2-preview.4`). Auth scope: `499b84ac-1321-427f-aa17-267ca6975798/.default` (the Azure DevOps resource app ID with the required OAuth2 v2.0 `/.default` suffix; the bare GUID is rejected by the Azure CLI with `AADSTS65002`).
- Content types: `application/json`, `application/json-patch+json` (work item create/update), `application/octet-stream` (binary).
- Work-item fetching is batched (200 per batch, 1000 max).
- Build request URLs safely — no user-controlled format strings that could lead to URL injection; URL-encode user-provided path/query segments.
- Azure credentials (Bearer tokens) are NEVER logged at any level. There are ZERO exceptions.

## 7) Output Optimization for LLM Consumption — ABSOLUTE RULES

MCP tool responses are optimized for LLM token consumption — use the existing helpers, do NOT emit verbose raw JSON:

- **Compact JSON** (`src/compact_llm.rs`): strips quotes from keys/values, removes whitespace, escapes only newlines/carriage returns.
- **Work item simplification** (`simplify_work_item_json`): removes `_links`, `url`, `descriptor`, `imageUrl`; flattens `fields` to root; strips Azure DevOps field prefixes (`System.`, `Microsoft.VSTS.*`); converts HTML to plain text (`html2text`); abbreviates work item types.
- **CSV output** (`work_items_to_csv`, `board_columns_to_csv`): tabular data returned as CSV, sanitized via `csv_sanitize`.
- Prefer compact/CSV output where appropriate; flag verbose JSON that inflates token usage unnecessarily.
