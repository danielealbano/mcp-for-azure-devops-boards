# Project Reference

## Overview

**mcp-for-azure-devops-boards** — A Model Context Protocol (MCP) server for interacting with Azure DevOps Boards and Work Items. Exposes Azure DevOps operations as MCP tools consumable by LLM agents.

- **Language**: Rust (edition 2024)
- **Async runtime**: tokio (full features)
- **MCP framework**: rmcp 0.8.5
- **Repository**: `github.com/danielealbano/mcp-for-azure-devops-boards`
- **License**: MIT

## Configuration

All configuration via CLI flags (clap):

| Flag | Default | Description |
|---|---|---|
| `--server` | false | Run in HTTP server mode (default: stdio) |
| `--port` | 3000 | HTTP server port (only with `--server`) |
| `--install` | — | Install MCP server configuration for the specified client (claude-code, claude-desktop, cursor, vscode, codex, gemini-cli) |

Environment variables:
- `RUST_LOG` — controls log level (e.g. `RUST_LOG=debug`)
- `AZURE_TENANT_ID` / `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET` — configure the environment client-secret credential; it is used only when all three are set (if only some are set it is skipped with a warning)
- Azure credentials are resolved via an ordered credential chain: environment client-secret (`AZURE_TENANT_ID` / `AZURE_CLIENT_ID` / `AZURE_CLIENT_SECRET`) → Azure CLI → Azure Developer CLI (local development) → managed identity (production / Azure-hosted). The managed-identity attempt is bounded by a 2s timeout so its IMDS probe fails fast off Azure.

## Transport Modes

- **stdio** (default): MCP communication over stdin/stdout. Used by LLM agent integrations (e.g. Cursor, Claude Desktop).
- **HTTP** (`--server`): Streamable HTTP transport via hyper. Binds to `0.0.0.0:<port>`. Uses `rmcp::StreamableHttpService` with `LocalSessionManager`.

## Code Generation Pipeline

Tool registration uses a two-phase code generation approach:

1. **Compile-time**: `build.rs` scans `src/mcp/tools/` for `#[mcp_tool(name = "...", description = "...")]` attributes, extracts function signatures, and generates `generated_tools.rs` in `OUT_DIR` containing the `#[tool_router] impl AzureMcpServer` block.
2. **Proc-macro**: `mcp-tools-codegen` provides the `#[mcp_tool]` attribute macro that validates metadata and passes through the function unchanged. The actual routing is generated by `build.rs`.

## Azure DevOps API Integration

- Base URL: `https://dev.azure.com/{organization}/{project}/_apis/`
- VSSPS URL: `https://app.vssps.visualstudio.com/_apis/`
- API version: 7.1 (Comments API: 7.2-preview.4)
- Auth: Bearer token via the Azure credential chain (scope: `499b84ac-1321-427f-aa17-267ca6975798/.default` — the Azure DevOps resource app ID with the required OAuth2 v2.0 `/.default` suffix)
- Content types: `application/json`, `application/json-patch+json` (for work item create/update), `application/octet-stream` (binary)

## Output Optimization

MCP tool responses are optimized for LLM consumption:
- **Compact JSON** (`compact_llm.rs`): Strips quotes from keys/values, removes whitespace, escapes only newlines. Example: `{id:123,Title:My Task,Tags:tag1;tag2}`
- **Work item simplification** (`simplify_work_item_json`): Removes `_links`, `url`, `descriptor`, `imageUrl`; flattens `fields` to root; strips Azure DevOps field prefixes (`System.`, `Microsoft.VSTS.*`); converts HTML to plain text; abbreviates work item types to first letter.
- **CSV output**: Work items returned as CSV for tabular consumption.

## Conventions

### Rust Idioms
- Follow the [Rust API Guidelines](https://rust-lang.github.io/api-guidelines/).
- Prefer the type system to enforce invariants at compile time.
- Use `Result<T, E>` everywhere — never `.unwrap()` in production code.
- Use `?` for error propagation. Wrap errors with context via `thiserror` variants or `.map_err()`.
- Prefer `snake_case` for functions/variables, `PascalCase` for types/traits, `SCREAMING_SNAKE_CASE` for constants.
- Keep modules small and cohesive. Prefer flat module trees.
- Use `pub` visibility sparingly — export only what consumers need.

### Async Patterns
- All I/O functions are `async`. Use `tokio::spawn` for concurrent tasks.
- Shared state across tasks uses `Arc<T>`. The `AzureDevOpsClient` is wrapped in `Arc` inside `AzureMcpServer`.
- No blocking calls in async context — use `tokio::task::spawn_blocking` if needed.

### Error Handling
- Domain errors: `thiserror`-derived `AzureError` enum in `src/azure/client.rs`.
- MCP tool errors: Convert `AzureError` to `rmcp::ErrorData` via `.map_err()`.
- Application errors: `anyhow::Result` in `main()`.

### Logging
- Facade: `log` crate. Backend: `env_logger`.
- Controlled via `RUST_LOG` environment variable.
- Never log secrets, tokens, or PII.

### MCP Tool Pattern
Each tool follows this pattern:
1. Define `Args` struct with `Deserialize` + `JsonSchema` (schemars).
2. Annotate the async function with `#[mcp_tool(name = "...", description = "...")]`.
3. Function signature: `pub async fn tool_name(client: &(dyn AzureDevOpsApi + Send + Sync), args: ArgsType) -> Result<CallToolResult, McpError>`.
4. Convert domain errors to `McpError` via `.map_err()`.
5. Return `tool_text_success(content)` (from `support/tool_text_success.rs`) — this automatically prepends the anti-prompt-injection warning to all tool responses. Never use `CallToolResult::success(vec![Content::text(...)])` directly.
